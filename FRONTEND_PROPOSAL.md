# Smart Health - Frontend Engineering & Design Report

This report defines the frontend architecture, design system, and user experience for the **Smart Health Monitoring System**. Designed to complement the Go backend, this frontend leverages **React** and **Tailwind CSS** to provide a high-performance, medical-grade interface for both doctors and patients.

---

## 1. Design Language & Brand Identity

The interface follows a **Modern Medical (Med-Tech)** aesthetic: clean, professional, and high-trust.

### 1.1 Color Palette
- **Primary (Doctor-Centric)**: `Blue-600 (#2563eb)` — evokes reliability, clinical precision, and calm.
- **Secondary (Patient-Centric)**: `Indigo-600 (#4f46e5)` — warmer, more approachable for personal health.
- **Urgent/Alert**: `Red-500 (#ef4444)` — immediate attention for threshold breaches (Tachycardia, Hypoxia).
- **Surface**: `Slate-50 (#f8fafc)` — background for high contrast.
- **Glassmorphism**: `White/80 with backdrop-blur` — used for modern card overlays.

### 1.2 Typography
- **Primary Font**: `Inter` (Sans-serif)
- **Scales**:
  - Headings: `font-bold tracking-tight`
  - Body: `font-normal text-slate-600`
  - Data/Vitals: `font-mono` (for readability of numeric values)

## 2. Architecture: React Ecosystem

### 2.1 Core Frameworks
- **Vite**: For ultra-fast development and optimized production builds.
- **React Router 6**: For declarative routing and role-based protection.
- **Tailwind CSS**: Utility-first styling for rapid, consistent UI.
- **Lucide React**: Lightweight, consistent iconography.

### 2.2 State & Data Management
- **TanStack Query (React Query)**: 
  - Essential for handling server state (fetching vitals, reports).
  - Handles auto-polling for live sensor data (e.g., `refetchInterval: 5000`).
- **React Context API**: 
  - For global auth state (JWT, User Profile, Roles).

## 3. Page Breakdown & UX Design

### 3.1 Landing Page: "The Digital Front Door"
*Focus: Trust, Capability, and Hardware Integration.*

- **Hero**: 3D illustration or high-res photo of the ESP32 hardware alongside a clean dashboard mockup.
- **Hardware Integration Section**: Detail the sensors (BPM, SpO2, ECG, Temp) with micro-animations.
- **AI-in-the-Loop Section**: Explain how LLMs (Claude/GPT) assist doctors, not replace them.
- **Dual-Portal CTAs**: Large, clear buttons for "Doctor Login" and "Patient Access."

### 3.2 Login Page: "Secure & Seamless"
*Focus: Security and Role Clarity.*

- **Design**: Centered card with subtle shadow.
- **Role Toggle**: A clean switch or tab system to select between **Doctor** and **Patient**.
- **UX**: Persistent "Remember Me" and "Forgot Password" flows.
- **Validation**: Real-time feedback for email format and password strength.

### 3.3 Doctor Portal: "Clinical Efficiency"
*Focus: Prioritization and Data Visualization.*

- **Sidebar Navigation**: Dashboard, Pending Reviews, Patient Directory, Settings.
- **The "Urgency Queue"**: 
  - Top of the dashboard features a list of patients with `is_urgent: true`.
  - Color-coded badges for specific breaches (e.g., "High Temp", "Low SpO2").
- **Report Review Interface**:
  - **Left Pane**: AI-generated Markdown draft with an "Edit" mode.
  - **Right Pane**: Interactive charts (Recharts) showing the 24-hour vital trends.
  - **Footer**: Single-click "Approve & Finalize" button that triggers the Go backend's PDF generation.

### 3.4 Patient Portal: "Empowered Health"
*Focus: Simplicity and Reassurance.*

- **Live Status Card**: Big, bold display of current BPM and SpO2 with a "Last Sync: Just Now" indicator.
- **Health Journey**: A vertical timeline of approved medical reports.
- **PDF Download**: One-click access to the dual-copy PDF generated by the doctor.
- **AI Insights Summary**: A simplified, non-clinical summary of their health trends for easy understanding.

## 4. Technical Implementation Strategy

### 4.1 Role-Based Access Control (RBAC)
Implement a `ProtectedRoute` component to wrap the portals:
```javascript
const ProtectedRoute = ({ role, children }) => {
  const { user, token } = useAuth();
  if (!token) return <Navigate to="/login" />;
  if (user.role !== role) return <Navigate to="/unauthorized" />;
  return children;
};
```

### 4.2 Handling Real-time Telemetry
Since the backend receives telemetry from ESP32, the React frontend should use React Query's polling to update charts without page refreshes:
```javascript
const { data: vitals } = useQuery(['vitals', patientId], fetchVitals, {
  refetchInterval: 5000, // Sync with ESP32 transmission rate
});
```

### 4.3 Styling with Tailwind
Use custom utility classes for "Vital Cards":
```html
<div className="p-6 rounded-2xl bg-white/80 backdrop-blur-md border border-slate-200 shadow-sm hover:shadow-md transition-all">

    <span className="text-4xl font-bold text-blue-600">82</span>
    <span className="text-slate-400 font-medium">BPM</span>
  </div>
</div>
```

## 5. Summary of Backend Alignment
- **Auth**: Frontend sends `Authorization: Bearer <token>` to all `/api/v1` routes.
- **Readings**: Consumes `/readings/:patient_id` for dashboard charts.
- **Reports**: Uses `/reports/pending` for the Doctor's worklist and `/reports/patient/:id` for the Patient's history.
- **PDFs**: Directly links to `/api/v1/reports/:id/pdf` for downloads.
